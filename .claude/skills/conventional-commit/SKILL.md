# conventional-commit

This repo's `VERSION` and `CHANGELOG.md` are generated by `release-please` by
parsing commit message prefixes (`feat:`, `fix:`, `refactor:`, `docs:`,
`perf:`, `chore:`, `test:`, plus a `BREAKING CHANGE:` footer for majors) — see
the root `CLAUDE.md`. A commit that doesn't follow this format either gets
silently ignored by release-please's changelog generation or, worse, mangles
the version bump. Getting the message right is not a style preference here;
it's what keeps the release automation correct.

## Before drafting a message

Only commit when the user has actually asked for a commit — this skill
doesn't change that norm, it just makes the message right once you're
committing. Follow this repo's standard git safety practice (stage specific
files by name, never `git add -A`/`git add .`, use a heredoc for the message
body, never `--no-verify`, never amend unless asked). This skill covers
**choosing the type/scope/description and structuring the body**, not the
git-safety rules, which apply regardless of skill.

Run `git status` and `git diff --staged` (and `git diff` if nothing is staged
yet — this repo's own convention is to stage everything relevant before
composing the message) to see the actual change. Also skim `git log
--oneline -15` for this repo's recent style — scopes here are usually a
package/directory name (`attachmentapi`, `ci`, `deps`, `all`) or omitted for
repo-wide changes, not a generic word like `code` or `stuff`.

## Choosing the type

Pick the type that matches what the diff *does*, not how the user described
the request. A user asking to "clean up the error handling" might have
actually introduced a `fix` (if a bug is corrected) or a `refactor` (if
behavior is unchanged) — read the diff to tell which.

| Type | Use when | Version bump |
|---|---|---|
| `feat` | New capability added (new exported function, new request builder, new API surface) | minor |
| `fix` | A bug is corrected (wrong output, crash, incorrect state) | patch |
| `refactor` | Code reshaped with no behavior change (rename, extract helper, consolidate duplication) | none |
| `docs` | Only `.md` files, doc comments, or `Readme.md` changed | none |
| `perf` | Change made specifically to improve speed/memory, not just incidentally faster | patch |
| `chore` | Tooling, CI, dependency bumps, config — nothing that ships to SDK consumers | none |
| `test` | Only `_test.go` files changed, no production code | none |

If the diff mixes types (e.g. a bug fix that also adds a test), pick the type
that describes the *primary* intent — usually the one a consumer of the
CHANGELOG would care about (`fix` outranks `test` here). Don't split one
logical change into multiple commits just to get separate type prefixes
unless the user asks for that.

### Breaking changes

If the diff changes an exported function signature, removes/renames an
exported identifier, or changes the meaning of existing behavior in a way
that could break a caller (see the CLAUDE.md warning about error-message text
changes breaking `errors.Is`/string-matching callers), add a
`BREAKING CHANGE:` footer describing what breaks and, if there's one, the
migration path. This is what tells release-please to cut a major version —
don't add it speculatively for internal-only (`internal/`) changes that
external consumers can't reach.

## Choosing the scope

Look at which package(s) the diff touches:

- Single package changed (`tableapi/`, `errors/`, `internal/store/`) → use
  that directory's base name as the scope, e.g. `fix(tableapi): ...`.
- Changes spread across many packages toward one goal → use `all` (matches
  this repo's existing usage, e.g. `refactor(all): error standardization`).
- CI/workflow files only (`.github/workflows/`) → scope `ci`.
- `go.mod`/`go.sum` dependency bumps → scope `deps`.
- Genuinely repo-wide or doesn't fit a single package → omit the scope
  entirely (`fix: merge errors`), don't force one.

## Writing the description

- Imperative mood, present tense: "add", not "added" or "adds".
- Lowercase after the colon, no trailing period.
- State *what changed*, not *why* — the reasoning belongs in the body if it's
  non-obvious, not the subject line.
- Keep the subject line under ~72 characters so it renders cleanly in the
  CHANGELOG.

## Body (optional but encouraged for non-trivial changes)

Add a body when the subject line alone wouldn't tell a future reader why the
change happened — a one-line `fix: correct nil check in GetHeaders` doesn't
need a body, but a multi-file refactor or a fix for a subtle bug does. Keep it
to 1-3 sentences focused on the *why*, matching this repo's general
commit-message guidance (see the root `CLAUDE.md` and system-level git
instructions): explain motivation, not a restatement of the diff.

## Assembling the final message

```
<type>[(<scope>)]: <description>

[optional body]

[optional BREAKING CHANGE: <description>]
```

Pass it via a heredoc, as usual for this repo:

```bash
git commit -m "$(cat <<'EOF'
fix(tableapi): correct nil check in GetHeaders

Prevented a panic when GetHeaders() was called on a table item
built without a request adapter.
EOF
)"
```

## Examples

**Input diff**: adds a new `Patch` method + request builder to `documentsapi`.
**Output**: `feat(documentsapi): add Patch request builder`

**Input diff**: `errors/errors.go` sentinel messages reworded, several
`*api` packages' nil-guards migrated to use them.
**Output**:
```
refactor(all): migrate nil-guards to shared error sentinels

BREAKING CHANGE: several sentinel error messages changed wording
(e.g. "request adapter is nil" -> "requestAdapter cannot be nil").
Callers doing err.Error() string comparison instead of errors.Is
will need to update.
```

**Input diff**: only `tableapi/Readme.md` edited to fix a broken link.
**Output**: `docs(tableapi): fix broken link in readme`

**Input diff**: `.github/workflows/ci.yml` bumps an action version.
**Output**: `chore(ci): bump actions/checkout to v7`
